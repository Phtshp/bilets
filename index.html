Билет №1

OSPF (Open Shortest Path First) — протокол динамической маршрутизации типа Link-State класса IGP (внутренний шлюз). Предназначен для работы внутри одной Автономной Системы (AS).

  • Принцип: Строит полную карту сети (LSDB) на каждом роутере. Кратчайшие пути вычисляются по алгоритму Дейкстры (SPF), где метрикой является Cost (обратно пропорционален пропускной способности). Сеть сходится за секунды.
  • Применение: Корпоративные сети, кампусы, инфраструктура внутри одного ЦОД.

BGP (Border Gateway Protocol) — протокол типа Path-Vector класса EGP (внешний шлюз). На нем функционирует глобальный Интернет.

  • Принцип: Маршрутизирует потоки данных между Автономными Системами. Оперирует не техническими скоростями каналов, а гибкими политиками и атрибутами (AS_PATH, Local Preference, MED, Weight). Сходимость медленная (до нескольких минут).
  • Применение: Стыки между интернет-провайдерами (ISP), подключение крупных организаций к нескольким операторам связи (Multihoming).

Конфигурация WireGuard описывается в файле /etc/wireguard/wg0.conf на обоих узлах. Обмен ключами происходит по асимметричной схеме (криптография Noise).

Обязательные параметры секции [Interface] (Локальные):
  • PrivateKey — закрытый ключ локального хоста (генерируется через wg genkey).
  • Address — IP-адрес интерфейса внутри туннеля с маской подсети (например, 10.0.0.1/24).
  • ListenPort — UDP-порт для приема пакетов (по умолчанию 51820).
Обязательные параметры секции [Peer] (Удаленный пир):
  • PublicKey — открытый ключ удаленной стороны (генерируется из его приватного ключа через wg pubkey).
  • AllowedIPs — список подсетей, трафик к которым заворачивается в туннель (например, 10.0.0.2/32, 192.168.1.0/24). Также выступает в роли криптографического фильтра.
  • Endpoint — внешний публичный IP-адрес и порт удаленного хоста (IP:порт). Необходим на инициирующей стороне.

VLAN (Virtual Local Area Network, IEEE 802.1Q) — технология логического разделения одной физической сетевой инфраструктуры (коммутатора L2) на множество изолированных широковещательных доменов.

Области применения:
  • Безопасность: Разделение сетей различных отделов (бухгалтерия, гостевой Wi-Fi, менеджеры). Узлы из разных VLAN не могут общаться на уровне L2 без маршрутизатора.
  • Локализация Broadcast-штормов: Ограничение распространения широковещательных пакетов рамками одного VLAN, что разгружает процессоры сетевых карт и коммутаторов.
  • Гибкость: Возможность логически объединять сотрудников в одну подсеть, даже если они сидят на разных этажах или в разных зданиях.

Билет №2

При поиске IP-адреса клиент шлет широковещательный запрос DHCPDISCOVER на адрес 255.255.255.255.

Почему не проходят: Маршрутизаторы по умолчанию отсекают и уничтожают широковещательный трафик (Broadcast), локализуя его в пределах одного сегмента/VLAN. Если DHCP-сервер находится за маршрутизатором в другой подсети, он этот запрос никогда не увидит.

Принцип работы DHCP Relay: На интерфейсе роутера, обслуживающем подсеть клиентов, включается агент ретрансляции (например, ip helper-address в Cisco). Роутер перехватывает бродкаст-запрос клиента, перепаковывает его в адресный пакет (Unicast), указывает в качестве Source IP свой адрес в этой подсети (giaddr) и шлет напрямую на IP-адрес удаленного DHCP-сервера. Сервер по полю giaddr понимает, из какого пула выдать адрес, и возвращает Unicast-ответ обратно на роутер.

1. Настройка сервера (в /etc/rsyslog.conf):

Необходимо активировать прием логов по сети, раскомментировав строки:

module(load="imudp")
input(type="imudp" port="514")
# или для TCP:
module(load="imtcp")
input(type="imtcp" port="514")
2. Настройка клиента (в /etc/rsyslog.conf):

В конец файла добавляется директива пересылки всех логов на удаленный сервер:

*.* @192.168.1.100:514   # @ означает протокол UDP
*.* @@192.168.1.100:514  # @@ означает протокол TCP

После правок на обоих узлах выполняется: systemctl restart rsyslog.

NTP (Network Time Protocol) — сетевой протокол (порт 123 UDP) для синхронизации внутренних системных часов серверов и оборудования с эталонными источниками точного времени (Stratum).

Зачем нужна синхронизация:
  • Корректность логов: Без единого времени невозможно сопоставить события (SIEM) с разных серверов при расследовании инцидентов.
  • Протоколы безопасности: Аутентификация Kerberos (FreeIPA, Active Directory) полностью блокируется, если разница времени между клиентом и контроллером домена составляет более 5 минут (защита от атак повтора).
  • Базы данных: Синхронизация критична для хронологии распределенных транзакций.

Билет №3

NAT (Network Address Translation) — механизм подмены IP-адресов в заголовках пакетов на маршрутизаторе во время их транзита.

  • SNAT (Source NAT): Модифицирует IP-адрес источника. Частный (серый) IP заменяется на публичный (белый) адрес роутера. Сценарий: обеспечение выхода сотрудников офиса из локальной сети в Интернет. Вариант с динамической подменой портов называется PAT или Маскарадинг (Masquerade).
  • DNAT (Destination NAT): Модифицирует IP-адрес назначения (и часто порт). Известен как «проброс портов» (Port Forwarding). Сценарий: предоставление доступа внешним пользователям из Интернета к внутреннему веб- или почтовому серверу компании, находящемуся за NAT в приватной сети.
Использование systemctl:
  • systemctl status sshd — выводит подробную информацию: статус (Active/Inactive/Failed), uptime, PID основного процесса, дерево процессов CGroup и последние строки лога.
  • systemctl is-active sshd — возвращает краткий статус (active/inactive), используется в скриптах автоматизации.
Использование journalctl:
  • journalctl -u sshd — выводит весь накопленный журнал системных логов конкретного сервиса.
  • journalctl -u sshd -f — режим реального времени ("живой" лог, прокрутка новых событий на экране).
  • journalctl -u sshd -p err — фильтрация записей, вывод только ошибок (уровень важности Error и выше).

SSH (Secure Shell) — защищенный криптографический сетевой протокол прикладного уровня, работающий по умолчанию на порту 22 TCP. Обеспечивает безопасное шифрование всего трафика, включая аутентификационные данные.

Применение:
  • Удаленное управление операционными системами Linux/Unix через интерфейс командной строки (CLI).
  • Безопасная передача файлов с помощью встроенных подсистем SFTP и утилиты SCP.
  • Создание зашифрованных SSH-туннелей для безопасной пересылки трафика сторонних приложений (например, проброс СУБД портов).

Билет №4

FreeIPA — централизованная система управления идентификацией, пользователями и политиками доступа в Linux-инфраструктуре (open-source аналог Active Directory).

Её компоненты:
  • LDAP (сервер 389-ds): Центральная база данных проекта. Хранит иерархическое дерево всех объектов: пользователей, хэшей паролей, групп, хостов, Sudo-правил и HBAC-политик.
  • Kerberos (KDC): Служба строгой аутентификации. Обеспечивает технологию Single Sign-On (SSO): пользователь получает один билет (TGT) при входе и далее прозрачно авторизуется на серверах без повторного ввода пароля.
  • DNS (BIND): Отвечает за сопоставление имен и обнаружение инфраструктурных сервисов домена с помощью автоматической генерации служебных записей SRV.
  • CA (Dogtag): Встроенный Центр Сертификации. Управляет жизненным циклом (выпуск, отзыв) сертификатов X.509 для обеспечения TLS-защиты сетевых служб домена.
  1. Генерация ключей на клиентском ПК: ssh-keygen -t ed25519. Будет создана пара: приватный id_ed25519 (остается в секрете) и публичный id_ed25519.pub.
  2. Перенос публичного ключа на сервер: ssh-copy-id -i ~/.ssh/id_ed25519.pub user@server_ip. Скрипт подключится к серверу и допишет ваш публичный ключ в файл ~/.ssh/authorized_keys в домашней директории целевого пользователя.
  3. Отключение паролей (Hardening): На сервере в файле /etc/ssh/sshd_config выставить директиву PasswordAuthentication no и выполнить systemctl restart sshd.

A-запись (Address Record) — тип ресурсной записи в системе доменных имен DNS, которая связывает символьное доменное имя хоста (например, test.ru) с его физическим 32-битным IPv4-адресом (например, 192.168.1.15). Это базовый элемент, необходимый для того, чтобы браузер или сервис понимали, на какой сетевой адрес слать IP-пакеты.

Билет №5

  • Авторитативный DNS-сервер: Является законным держателем и первоисточником информации о конкретной доменной зоне (например, содержит файлы описания зоны company.local). Если его спросить о записи внутри его зоны — он выдаст точный авторитативный ответ. Чужие домены он искать в Интернете не умеет.
  • Рекурсивный DNS-резолвер: Сервер-посредник, обслуживающий запросы конечных клиентов (например, 8.8.8.8). Он сам не хранит зоны, но умеет самостоятельно проходить всю глобальную цепочку иерархии DNS (Корневые серверы -> TLD-серверы верхнего уровня -> Авторитативные серверы), чтобы найти нужный IP-адрес для клиента, после чего кэширует его для ускорения будущих ответов.
  • dig example.com — базовый запрос A-записи. Статус NOERROR и наличие записей в ANSWER SECTION говорят об успешной работе.
  • dig @192.168.1.10 example.com MX — запрос почтовых (MX) записей домена напрямую у конкретного тестируемого DNS-сервера (в обход кэша).
  • dig -x 192.168.1.50 — проверка обратной зоны (записи PTR), сопоставляющей IP с доменным именем.
  • nslookup -type=any example.com — вывод всех существующих типов записей для домена через nslookup.

Доменное имя — удобочитаемое алфавитно-цифровое имя, служащее для идентификации сетевых узлов и ресурсов в Интернете. Имеет строгую иерархическую структуру, элементы которой разделены точками (читается справа налево): корневая зона (.), домены первого уровня TLD (.com, .ru), второго уровня (yandex.ru) и поддомены третьего уровня (mail.yandex.ru).

Билет №6

VRRP (Virtual Router Redundancy Protocol) объединяет несколько физических маршрутизаторов в одну логическую группу с общим виртуальным IP (VIP) и виртуальным MAC-адресом. Именно этот VIP настраивается на хостах сети как шлюз по умолчанию.

  • Среди роутеров на основе приоритета выбирается один MASTER, остальные становятся BACKUP.
  • MASTER присваивает себе виртуальный IP и обрабатывает весь трафик пользователей. Раз в секунду он шлет мультикаст-сообщения (на адрес 224.0.0.18), подтверждая свой статус.
  • Если MASTER пропадает, BACKUP роутеры фиксируют отсутствие объявлений. Спустя тайм-аут (~3 сек) BACKUP с наивысшим приоритетом объявляет себя новым MASTER, шлет Gratuitous ARP в сеть, перехватывает VIP на свой физический порт и продолжает маршрутизацию. Пользователи сбоя не замечают.

Утилита rsync эффективно синхронизирует директории, передавая только измененные части файлов (дельта-алгоритм).

Типичная команда для локального или сетевого бэкапа:

rsync -avz --delete /var/www/html/ /mnt/backup/web_mirror/
Разбор ключей:
  • -a (archive): Архивный режим. Сохраняет права доступа, владельцев, временные метки и вложенные символические ссылки. Включает рекурсивный обход.
  • -v (verbose): Выводит детальный лог работы в консоль.
  • -z (compress): Сжимает данные на лету при передаче.
  • --delete: Удаляет файлы в целевой папке бэкапа, если они были удалены в исходном каталоге (поддерживает точное зеркало).
  • Полное копирование (Full Backup): Копирует абсолютно весь объем выбранных данных в архив. Плюсы: максимальная простота восстановления (нужен один архив). Минусы: долгий процесс создания, занимает огромный объем диска.
  • Инкрементальное копирование (Incremental Backup): Архивирует только те файлы, которые были изменены или созданы с момента проведения любого последнего бэкапа (полного или инкрементального). Плюсы: выполняется моментально, экономит дисковое пространство. Минусы: медленное и сложное восстановление (требуется развернуть базовый полный бэкап и поочередно накатить всю цепочку последующих инкрементов).

Билет №7

  • WireGuard: Лидер по производительности. Работает внутри ядра Linux, использует современную легковесную криптографию (ChaCha20-Poly1305). Конфигурация чрезвычайно простая, кодовая база минимальна (легко аудировать). Минус: из коробки работает только по UDP.
  • OpenVPN: Средняя производительность, так как работает в пространстве пользователя (User space) и постоянно гоняет данные через виртуальный интерфейс TUN/TAP, загружая процессор переключениями контекста. Очень гибкий в настройке (TCP/UDP, любые порты, обход файрволов), сложная конфигурация на базе сертификатов OpenSSL.
  • IPsec: Высокая скорость (также реализован на уровне ядра). Индустриальный стандарт. Имеет тяжеловесную и крайне сложную архитектуру настроек (IKEv1/IKEv2, фазы шифрования, политики безопасности). Сложно настраивать вручную, часто применяется для Site-to-Site туннелей между корпоративными роутерами.
  • wg show — главная диагностическая команда. Показывает статус интерфейса, открытые ключи, IP-адреса пиров и, самое главное, строку latest handshake (время последнего успешного обмена пакетами приветствия) и счетчик трафика (transfer). Если handshake есть — туннель активен.
  • ip address show wg0 — проверка того, поднялся ли сетевой интерфейс и присвоен ли ему корректный внутренний IP-адрес.
  • ping 10.0.0.2 — проверка базовой связности, отправка ICMP-запроса на внутренний IP-адрес удаленного пира.

VPN (Virtual Private Network) — технология создания защищенного, зашифрованного логического сетевого соединения (туннеля) поверх незащищенной публичной сети (Интернета). Обеспечивает конфиденциальность, целостность данных и защиту от перехвата трафика сторонними лицами.

Билет №8

Оба протокола работают на транспортном уровне модели OSI, но решают разные задачи.

  • TCP (Transmission Control Protocol): Протокол с установлением соединения (Three-way handshake). Гарантирует доставку пакетов, контролирует их строгую очередность, управляет потоком данных для исключения потерь, делает повторный запрос при утере. Применение: веб-страницы (HTTP/HTTPS), почта (SMTP/IMAP), удаленный доступ (SSH), СУБД — там, где критична целостность данных.
  • UDP (User Datagram Protocol): Протокол без установления соединения. Отправляет датаграммы "как есть", не проверяя, дошли они до получателя или нет, и не следит за порядком. За счет отсутствия накладных расходов обеспечивает максимальную скорость и минимальные задержки. Применение: потоковое видео/аудио, IP-телефония (VoIP), онлайн-игры, DNS, NTP.

В верхней панели дисплейных фильтров Wireshark необходимо ввести следующее логическое выражение:

ip.addr == 192.168.1.15 && ip.addr == 192.168.1.25

Этот фильтр оставит на экране только те пакеты, у которых в полях Source или Destination фигурируют оба указанных IP-адреса. Для изоляции конкретного типа протокола (например, только веб-трафик между ними) выражение можно расширить: ip.addr == 192.168.1.15 && ip.addr == 192.168.1.25 && tcp.port == 443.

Порт — это 16-битное числовое значение (от 0 до 65535), записываемое в заголовок транспортного уровня. Используется операционной системой для адресации сетевых пакетов конкретному приложению или службе, запущенной на хосте (например, порт 80 для веб-сервера, порт 22 для SSH).

Билет №9

QoS (Quality of Service) — набор технологий управления ресурсами сети, предназначенный для приоритезации определенных типов трафика (голос, видео) за счет других (скачивание файлов, почта).

Влияние сетевых характеристик на голос (VoIP):

Голосовой трафик критичен к задержкам, так как общение происходит интерактивно в реальном времени.

  • Задержка (Delay): Односторонняя задержка пакетов не должна превышать 150 мс. При больших значениях общение превращается в режим рации, собеседники начинают перебивать друг друга.
  • Джиттер (Jitter): Колебание задержки между пакетами. Если пакеты приходят неравномерно, звук становится прерывистым, "роботизированным". Для компенсации применяется джиттер-буфер.
  • Потери пакетов (Packet Loss): Потеря более 1–2% голосовых пакетов приводит к выпадению слогов и слов из разговора, делая речь неразборчивой.

Тестирование требует участия двух узлов в сети.

  1. На первом сервере (Приемник/Сервер): Запускается iperf3 в режиме ожидания подключений:
    iperf3 -s
  2. На втором сервере (Отправитель/Клиент): Запускается генерация трафика в сторону первого сервера:
    iperf3 -c 192.168.1.100 -t 10 -P 4

Параметр -t 10 задает время теста (10 секунд), а -P 4 запускает тест в 4 параллельных потока, что позволяет выжать максимум из сетевого канала. По окончании клиент и сервер выведут итоговую скорость в Мбит/с (Bandwidth).

RTP (Real-time Transport Protocol) — специализированный сетевой протокол прикладного уровня, работающий поверх UDP. Используется для непосредственной передачи медиаданных (аудиопотоков VoIP, видеотрансляций). Он снабжает каждый пакет порядковым номером и временной меткой, позволяя принимающей стороне собрать поток в правильном порядке и воспроизвести его без искажений времени.

Билет №10

  • Apache HTTP Server: Использует классическую архитектуру (MPM Prefork/Worker), где под каждое входящее клиентское TCP-соединение выделяется отдельный независимый процесс или поток ОС. При высоком Concurrency (тысячи клиентов) система упирается в дефицит RAM и тратит ресурсы процессора на бесконечное переключение контекста между потоками.
  • Nginx: Построен на базе асинхронного событийно-ориентированного неблокирующего цикла (Event Loop) с вызовами epoll/kqueue. Фиксированное число рабочих процессов (Worker) в один поток обрабатывает сотни тысяч соединений параллельно по мере поступления событий ввода-вывода.
Эффективность Nginx со статикой:

При запросе статических файлов (css, html, картинок) Nginx задействует системный вызов ядра Linux sendfile. Данные копируются напрямую из файлового дескриптора диска в сетевой сокет внутри пространства ядра (Kernel Space), минуя копирование в пространство пользователя (User Space). Это минимизирует нагрузку на CPU и позволяет отдавать файлы на максимальной скорости дисковой подсистемы при минимальном потреблении ОЗУ.

Изолированная конфигурация для конкретного сайта описывается в блоке server внутри директории /etc/nginx/sites-available/:

server {
    listen 80;
    server_name mywebsite.ru www.mywebsite.ru;

    root /var/www/mywebsite;
    index index.html index.htm;

    location / {
        try_files $uri $uri/ =404;
    }

    access_log /var/log/nginx/mywebsite_access.log;
    error_log /var/log/nginx/mywebsite_error.log;
}

После создания делается симлинк в папку sites-enabled, проверяется синтаксис: nginx -t, и перезапускается конфигурация: systemctl reload nginx.

Стандартный незащищенный веб-протокол HTTP функционирует по умолчанию на порту 80 TCP. Защищенный протокол HTTPS, использующий криптографическое шифрование TLS/SSL для защиты сессий, работает на порту 443 TCP.

Билет №11

Stateful Firewall (межсетевой экран с контролем состояний) анализирует пакеты не изолированно, а в контексте сетевых сессий. Он содержит динамическую таблицу соединений (в Linux это механизм conntrack), где регистрирует параметры открытых сессий (IP источника/назначения, порты, флаги TCP).

Когда внутренний пользователь инициирует запрос к внешнему сайту, файрвол заносит эту сессию в таблицу. Когда от сайта приходит входящий ответный пакет, файрвол сверяет его с таблицей conntrack, идентифицирует его статус как ESTABLISHED (соединение установлено) или RELATED (связанное) и автоматически пропускает пакет внутрь сети, даже если входящие соединения на этот порт глобально заблокированы правилами. Это избавляет от необходимости вручную открывать порты для обратного трафика.

В современной подсистеме фильтрации Linux nftables правило добавляется в соответствующую цепочку (обычно input) вашей таблицы:

nft add rule ip filter input ip saddr 192.168.10.0/24 tcp dport 22 accept

Разбор правила: ip filter input — указание таблицы и цепочки; ip saddr 192.168.10.0/24 — фильтр по подсети источника (Source Address); tcp dport 22 — фильтр транспортного протокола и целевого порта SSH; accept — вердикт (разрешить пакет).

  • DROP: Пакет полностью игнорируется и молча уничтожается файрволом. Отправитель не получает никакого уведомления, его сетевой стек ждет ответа до истечения системного тайм-аута (Timeout). Применяется на внешних интерфейсах, чтобы скрыть присутствие хоста в сети от сканеров злоумышленников.
  • REJECT: Пакет блокируется, но файрвол отправляет обратно автору официальное уведомление об ошибке. Для UDP/ICMP шлется пакет ICMP Port Unreachable, для TCP — пакет с флагом TCP RST (Reset). Клиентское приложение мгновенно узнает, что доступ закрыт. Применяется внутри доверенных локальных сетей для ускорения отладки.

Билет №12

Назначение VLAN — логическое структурирование и изоляция подсетей внутри единой физической коммутационной фабрики. Влияние на широковещательные домены является ключевым свойством: VLAN разбивает один физический широковещательный домен (Flat Network) на несколько независимых изолированных L2-сегментов.

Широковещательный кадр (например, ARP-запрос), сгенерированный хостом в VLAN 10, будет размножен коммутатором только на те порты, которые также привязаны к VLAN 10. Устройства в VLAN 20 этот кадр никогда не получат. Это экономит ресурсы полосы пропускания и защищает от перегрузок.

Trunk (магистральный порт) передает трафик сразу нескольких VLAN. Для идентификации принадлежности кадров к каждому пакету добавляется специальный 4-байтовый тег 802.1Q, содержащий маркер VLAN ID (VID).

Пример настройки интерфейса на коммутаторе Cisco CLI:

interface gigabitEthernet 0/1
 description Uplink_to_Switch_B
 switchport mode trunk                   # Перевод порта в режим транка
 switchport trunk allowed vlan 10,20,30  # Список разрешенных к пропуску VLAN

Access (доступный/доступа) порт — порт коммутатора, который жестко привязан к одному конкретному VLAN (Native VLAN). В него включаются конечные потребители (ПК, серверы, принтеры), сетевые карты которых не умеют обрабатывать теги 802.1Q. При выходе пакета из access-порта наружу к компьютеру, коммутатор автоматически удаляет тег, делая кадр стандартным.

Билет №13

  • Одномодовое волокно (SM - Singlemode): Обладает очень тонким диаметром светопроводящей сердцевины (около 9 мкм). Внутри него распространяется только один луч (мода) света. За счет отсутствия межмодовой дисперсии сигнал практически не искажается. Дальность передачи — до десятков и сотен километров. В качестве источников света используются дорогие, высокоточные полупроводниковые лазеры.
  • Многомодовое волокно (MM - Multimode): Имеет широкую сердцевину (50 или 62.5 мкм). Световой импульс распространяется одновременно по множеству траекторий (мод). Из-за этого возникает межмодовая дисперсия (лучи приходят к приемнику с разным опозданием), что сильно ограничивает дальность передачи (обычно до 300–500 метров). Использует дешевые светодиоды (LED) или вертикально-излучающие лазеры (VCSEL). Применяется внутри стоек ЦОД.
  • Метод LSM (измеритель мощности и источник излучения): На один конец кабеля ставится эталонный излучатель стабильной мощности, на другой — калиброванный измеритель (Optical Power Meter). Сравнивая отправленную и полученную мощность, вычисляют точные сквозные потери линии в децибелах (дБ).
  • Метод оптической рефлектометрии (OTDR): Прибор посылает в линию мощные короткие световые импульсы и анализирует уровень обратно рассеянного отраженного сигнала. Рефлектометр строит график (трассу), на которой наглядно видны все локальные неоднородности: места сварок, разъемы, макроизгибы волокна и точное расстояние до места обрыва в метрах.

Это оптическое явление, происходящее при переходе светового луча из оптически более плотной среды (стеклянная сердцевина волокна с высоким показателем преломления $n_1$) в менее плотную (оболочка кабеля с меньшим показателем $n_2$). Если угол падения луча превышает определенный критический уровень, луч не преломляется наружу, а целиком отражается обратно внутрь ядра. Благодаря этому свет зажимается внутри кабеля и распространяется по нему как по волноводу.

Билет №14

RAID (Redundant Array of Independent Disks) — технология виртуализации данных, объединяющая несколько физических жестких дисков в один логический элемент для достижения отказоустойчивости и/или ускорения чтения-записи.

  • RAID 1 (Зеркалирование / Mirroring): Данные дублируются и параллельно пишутся на два диска. Доступная емкость равна объему одного диска. Скорость чтения удваивается, скорость записи неизменна. Выдерживает смерть 1 накопителя.
  • RAID 5 (Чередование с контролем четности): Требует минимум 3 диска. Данные режутся на блоки и циклически пишутся на диски вперемешку с контрольными суммами четности (XOR). Полезный объем равен $(N-1)$. Выдерживает потерю любого 1 диска. Обладает штрафом на запись из-за пересчета сумм.
  • RAID 10 (1+0 - Зеркало из страйпов): Требует минимум 4 диска. Объединяет скорость чередования (RAID 0) и надежность зеркалирования (RAID 1). Обладает максимальной производительностью. Доступный объем — половина от суммарного. Может пережить одновременный отказ до 2 дисков (если они из разных зеркальных пар).

Для создания софт-массивов применяется системная утилита mdadm. Создадим зеркало (RAID 1) из двух свободных накопителей /dev/sdb и /dev/sdc:

mdadm --create --verbose /dev/md0 --level=1 --raid-devices=2 /dev/sdb /dev/sdc

После создания можно отслеживать ход синхронизации дисков через файл конфигурации ядра: cat /proc/mdstat. Затем на созданном виртуальном устройстве /dev/md0 создается файловая система: mkfs.ext4 /dev/md0 и диск готов к монтированию.

RAID применяется для защиты инфраструктуры от потери данных и простоев при физическом выходе из строя жестких дисков (обеспечение непрерывности бизнеса), а также для ликвидации узких мест производительности дисковой подсистемы (I/O) путем распараллеливания потоков данных.

Билет №15

Протокол TLS (Transport Layer Security) обеспечивает защиту данных на базе гибридного шифрования.

Принцип работы (Handshake):
  1. Клиент и сервер согласуют поддерживаемые алгоритмы (Cipher Suite).
  2. Сервер присылает свой цифровой сертификат X.509, содержащий его открытый ключ.
  3. Происходит асимметричный обмен ключами (например, ECDHE), на основе которого генерируется общий сессионный ключ.
  4. Вся дальнейшая пересылка шифруется уже симметрично этим ключом (AES/CHACHA) ради экономии ресурсов CPU.
Проверка подлинности (Аутентификация сервера):

Браузер (клиент) проверяет пришедший сертификат по цепочке доверия. Он извлекает цифровую подпись сертификата и расшифровывает её встроенными в ОС открытыми ключами корневых Удостоверяющих Центров (CA). Если подпись сходится, доменное имя в сертификате совпадает с адресом в строке браузера, а срок действия не истек — сервер считается легитимным, и защита устанавливается.

Запрос CSR генерируется с помощью пакета openssl. Команда одновременно создает приватный ключ сервера и сам файл запроса:

openssl req -new -newkey rsa:2048 -nodes -keyout mydomain.key -out mydomain.csr

В процессе выполнения утилита интерактивно потребует заполнить метаданные компании: код страны (RU), название организации, и главное поле — Common Name (CN), куда вписывается точный домен сайта (например, mydomain.ru). Полученный файл .csr передается в CA на подпись.

Это общепринятый строгий международный стандарт для цифровых сертификатов публичных ключей. Он представляет собой структурированный электронный документ, завереный электронной подписью доверенного Центра Сертификации, который гарантирует, что конкретный открытый ключ принадлежит строго определенному владельцу (домену/серверу).

Билет №16

Маршрутизация в OSPF базируется на знании полной картины сети всеми участниками.

Формирование базы данных LSDB:

После установления соседства (через Hello-пакеты) маршрутизаторы начинают генерировать пакеты LSA (Link-State Advertisement), в которых описывают свои активные интерфейсы, подключенные подсети и стоимость каналов (метрику). Роутеры лавинообразно обмениваются этими LSA со всеми соседями в рамках одной зоны (Area). Каждый роутер складывает все полученные LSA в единую локальную базу данных — LSDB (Link-State Database). Благодаря механизмам синхронизации, базы LSDB на всех маршрутизаторах зоны становятся абсолютно идентичными. На основе этой базы алгоритм SPF рассчитывает оптимальное дерево маршрутов.

Для просмотра текущей таблицы маршрутизации ядра Linux используется современная утилита из пакета iproute2:

ip route show (или сокращенно ip r)

Каждая строка вывода описывает целевую сеть, через какой сетевой интерфейс (dev) отправлять пакет, IP-адрес следующего хопа (via) и метрику маршрута (сортировка приоритета).

Это специальный маршрут, обозначаемый в сетевых настройках как 0.0.0.0/0 (в нотации IPv6 — ::/0). Он является "маршрутом последней надежды". Если в таблице маршрутизации роутера не нашлось точного совпадения для IP-адреса получателя пакета, пакет пересылается по этому дефолтному пути (на адрес вышестоящего провайдера).

Билет №17

Классический протокол NFS (v3/v4) по умолчанию небезопасен — он доверяет числовым идентификаторам UID/GID, которые присылает клиент в заголовках пакетов (их легко подделать на локальной Linux-машине).

При интеграции NFSv4 с Kerberos (режимы krb5, krb5i, krb5p) архитектура безопасности меняется:

  • Перед монтированием сетевой папки клиентский хост и сервер обязаны пройти взаимную строгую криптографическую проверку подлинности в Kerberos KDC, получив сервисные тикеты nfs/fqdn@REALM.
  • Вместо слепого доверия UID, сервер авторизует пользователя на основе проверенного Kerberos-принципала.
  • В режиме krb5i каждый пакет подписывается хэшем для защиты от подмены, а в режиме krb5p — весь трафик файлов полностью шифруется внутри сетевого канала, защищая конфиденциальные данные от перехвата.
1. На стороне NFS-сервера:

В конфигурационный файл /etc/exports добавляется строка-права:

/opt/shared_data  192.168.1.0/24(rw,sync,no_subtree_check)

После этого применяется команда экспорта без рестарта демона: exportfs -arv, и запускается служба: systemctl enable --now nfs-server.

2. На стороне NFS-клиента:

Монтирование осуществляется стандартной системной командой:

mount -t nfs 192.168.1.10:/opt/shared_data /mnt/nfs_local/

NFS (Network File System) — популярный протокол сетевого доступа к файловым системам, позволяющий пользователям и программам монтировать удаленные директории сервера по сети и работать с ними точно так же, как если бы они располагались на локальном физическом диске машины.

Билет №18

Выдача IP-адреса хосту происходит по четырехэтапному алгоритму DORA на базе портов 67 и 68 UDP:

  • D - Discover (Обнаружение): Клиент шлет широковещательный пакет в сеть, заявляя: «Мне нужен IP-адрес».
  • O - Offer (Предложение): DHCP-сервер перехватывает пакет, выбирает из своего пула свободный IP, резервирует его по MAC-адресу клиента и шлет ответ: «Могу предложить IP 192.168.1.50».
  • R - Request (Запрос): Клиент получает предложение (или несколько) и официально отвечает бродкастом: «Отлично, я запрашиваю именно этот IP 192.168.1.50 у сервера А». Broadcat нужен, чтобы остальные серверы узнали, что их предложения отклонены.
  • A - Acknowledge (Подтверждение): Сервер фиксирует аренду в базе данных и отправляет финальный пакет, содержащий IP, маску, шлюз, адреса DNS и время аренды (Lease Time). Клиент применяет параметры.

Для корректной работы сетевого стека клиента в пуле раздачи обязательно прописываются:

  • range — диапазон (начальный и конечный IP), выделенный для динамической раздачи.
  • netmask — маска подсети, определяющая границы локального сегмента.
  • default gateway — IP-адрес маршрутизатора для выхода во внешние сети.
  • domain-name-servers (DNS) — IP-адреса резолверов для разрешения имен сайтов.
  • lease time — время аренды IP-адреса, по истечении половины которого клиент обязан запросить продление.

Шлюз по умолчанию (Default Gateway) — сетевое устройство (обычно L3-маршрутизатор), на которое сетевой стек компьютера отправляет все пакеты, адрес назначения которых не принадлежит текущей локальной подсети хоста. Шлюз знает, как доставить эти данные в другие внешние сети.

Билет №19

Архитектура состоит из трех уровней: Агенты сбора (Log Forwarders: Fluentbit, Filebeat, Logstash) на серверах -> Шина/Коллектор данных (Rsyslog, Vector, Kafka) -> Хранилище с UI (Elasticsearch, OpenSearch, Grafana Loki).

Преимущества централизованного сбора логов:
  • Быстрая отладка и аудит: Возможность выполнять сквозной поиск по ключевым словам (например, `id_пользователя`) сразу по всей инфраструктуре из единого окна, не заходя по SSH на сотни машин.
  • Живучесть данных: Если сервер полностью сгорит или его зашифрует вирус, логи его работы останутся в сохранности на сервере логов.
  • Безопасность: Злоумышленник, взломавший сервер, не сможет скрыть следы преступления, затерев локальные файлы в /var/log/, так как логи улетают на независимый коллектор мгновенно в режиме реального времени.

Для просмотра свежих логов без открытия полной истории применяется утилита с флагом количества строк -n:

journalctl -u nginx.service -n 50 --no-pager

Ключ -n 50 указывает вывести строго последние 50 записей из журнала, а флаг --no-pager отключает интерактивную утилиту less, печатая логи сразу в консоль, что удобно для копирования.

Системный лог (Log file) — специальный файл, куда ядро операционной системы, демоны и прикладные приложения ведут непрерывную последовательную хронологическую запись текстовых сообщений обо всех происходящих событиях, изменениях статусов, предупреждениях и критических ошибках.

Билет №20

  • Bluetooth (включая BLE): Работает на частоте 2.4 ГГц. Малый радиус действия (до 10–15 метров), средняя скорость передачи. Разработан для связи персональных пользовательских гаджетов со смартфонами (беспроводные наушники, фитнес-браслеты, мыши). Отличается высокой скоростью установки соединения.
  • Zigbee: Частота 2.4 ГГц или 868 МГц. Малый радиус действия одиночного узла (до 20-50 метров), низкая скорость передачи, минимальное энергопотребление. Главная особенность — поддержка Mesh-топологии (каждое устройство с постоянным питанием работает ретранслятором для соседей). Идеально подходит для систем автоматизации зданий и «Умного дома» (датчики протечки, температуры, умные розетки).
  • LoRaWAN (Long Range WAN): Субгигагерцевые безлицензионные частоты (868 МГц в РФ/Европе). Огромный радиус действия (до 5–15 км от базовой станции) при ничтожно малой скорости передачи (килобиты). Датчики могут работать от одной батарейки-таблетки до 5–10 лет. Предназначен для промышленного Интернета Вещей (IoT), телеметрии ЖКХ (счетчики воды/газа на кварталах) и умного сельского хозяйства.

Сеть Zigbee базируется на строгом разделении ролей устройств:

  • Координатор (Coordinator): Главное устройство в сети (мозг). Инициализирует сеть, распределяет адреса, хранит ключи шифрования. В сети всегда строго один координатор (обычно это шлюз умного дома).
  • Маршрутизаторы (Routers): Устройства, подключенные к постоянной сети 220V (умные выключатели, розетки). Они не только выполняют свои функции, но и постоянно транслируют через себя пакеты от дальних узлов к координатору, расширяя зону покрытия сети. Сеть является самоорганизующейся — при выходе одного роутера из строя, пакеты автоматически пойдут в обход через другие узлы.
  • Конечные устройства (End Devices): Спящие беспроводные датчики на батарейках. Они не участвуют в маршрутизации, большую часть времени спят для экономии энергии, просыпаясь только для передачи показаний своему родительскому роутеру.

BLE (Bluetooth Low Energy) — независимая ветка спецификации Bluetooth, оптимизированная для циклической передачи небольших объемов данных с ультранизким энергопотреблением, что позволяет миниатюрным IoT-устройствам годами функционировать без замены элементов питания.

Билет №21

Архитектурный упор в Wi-Fi 6 сделан не на пиковую скорость одного клиента, а на высокую стабильность и емкость радиоэфира в условиях плотной городской или офисной застройки (High Efficiency).

Ключевые технологии эффективности:
  • OFDMA (Orthogonal Frequency Division Multiple Access): Главная фишка стандарта. Позволяет разделять один широкий частотный канал на множество мелких подканалов — RU (Resource Units). Роутер может отправлять пакеты одновременно десятку разных клиентов в один и тот же квант времени, радикально снижая задержки (Latency) и убирая очереди в эфире.
  • MU-MIMO (Multi-User MIMO) в режимах Downlink и Uplink: Использование многоантенных систем для формирования пространственных лучей, что позволяет одновременно прокачивать тяжелые потоки данных на/от нескольких устройств параллельно.
  • BSS Coloring (Окрашивание спектра): Каждому роутеру присваивается цифровой маркер-цвет. Если точка слышит чужой сигнал на своей частоте, но с "другим цветом", она игнорирует его и не тратит время на ожидание освобождения эфира, преодолевая проблему интерференции в многоэтажках.
  • TWT (Target Wake Time): Роутер жестко согласовывает с клиентами (например, смартфонами) график их выхода на связь. Клиент знает точные миллисекунды, когда нужно включить радиомодуль, а всё остальное время полностью спит, экономя до 30-40% батареи.
  • Препятствия и материалы: Физические преграды поглощают и отражают радиоволны. Железобетонные стены, зеркала, тонированные стекла и металлоконструкции могут полностью погасить сигнал 5 ГГц.
  • Затухание с расстоянием (Free Space Path Loss): Сигнал высокой частоты затухает естественным образом по мере удаления от антенны. Частота 5 ГГц затухает быстрее, чем 2.4 ГГц.
  • Интерференция (Зашумленность эфира): Наличие соседних роутеров, работающих на тех же или перекрывающихся частотных каналах. Также диапазон 2.4 ГГц зашумляют бытовые СВЧ-печи, радионяни и Bluetooth-устройства.
  • Логические факторы: Неправильный выбор ширины канала (например, принудительные 80 МГц в зашумленном эфире ловят больше коллизий, чем стабильные 20 МГц) и неверное пространственное расположение антенн.

SSID (Service Set Identifier) — текстовое уникальное название беспроводной Wi-Fi сети длиной до 32 байт, транслируемое точкой доступа в эфир в специальных кадрах Beacon, служащее для идентификации и выбора нужной сети пользователем при подключении.

Билет №22

BGP (Border Gateway Protocol, текущая версия BGP-4) оперирует понятием Автономных Систем (AS). Обмен информацией идет по протоколу TCP (порт 179). Роутеры-соседи (BGP Peers) анонсируют друг другу списки сетевых префиксов, к которым у них есть доступ.

Роль атрибута AS_PATH:

AS_PATH — обязательный Well-known Mandatory атрибут BGP. Когда роутер передает анонс сети в соседнюю Автономную Систему, он дописывает номер своей собственной AS в начало цепочки этого атрибута. На выходе получается векторный список номеров AS, которые нужно пересечь пакету для достижения цели.

Он решает две важнейшие задачи:

  • Предотвращение петель маршрутизации (Loop Prevention): Если BGP-роутер получает от соседа анонс маршрута, в атрибуте AS_PATH которого уже значится номер его собственной Автономной Системы, он понимает, что пакет совершил круг, и мгновенно отбрасывает этот маршрут.
  • Выбор лучшего пути: При прочих равных условиях алгоритм BGP выбирает тот маршрут, у которого список AS_PATH короче (содержит меньше хопов по автономным системам).

Для диагностики статуса сессий используется классическая команда Cisco/Quagga/FRR CLI:

show ip bgp summary

В выводимой таблице критически важно смотреть на самый последний столбец — State/PfxRcd:

  • Если там написано текстовое значение (Active, Idle, Connect) — это означает, что BGP-сессия **не поднялась**, произошла ошибка конфигурации, пароля MD5 или блокировка файрволом.
  • Если там отображается **число** (например, 14500) — сессия установлена успешно (статус Established), а число показывает количество сетевых маршрутов (префиксов), принятых от данного соседа.

Автономная система (AS) — обширная совокупность IP-сетей и маршрутизаторов, находящихся под единым техническим и административным контролем одной организации (провайдер связи, банк, корпорация), использующая общую согласованную политику маршрутизации во внешнем мире и имеющая уникальный зарегистрированный номер ASN.

Билет №23

Ansible — декларативная система оркестрации и автоматизации ИТ-инфраструктуры.

Принцип работы:

Ansible исповедует **Agentless-архитектуру**. На целевые серверы не нужно ставить никакой софт или агенты. Администратор на своей рабочей станции пишет сценарий — **Playbook** на понятном языке разметки YAML. Ansible считывает сценарий, подключается к удаленным серверам по стандартному протоколу SSH, генерирует под каждую задачу временные скрипты-модули на языке Python, загружает их на сервер, выполняет и удаляет за собой.

Преимущества Ansible:
  • Идемпотентность (Idempotency): Ключевое свойство. Повторный запуск плейбука не приведет к дублированию настроек или ошибкам. Модули сначала проверяют текущее состояние сервера: если конфигурация уже соответствует описанной в сценарии, Ansible ничего делать не будет, а просто отрапортует `OK`.
  • Простота внедрения: Отсутствие агентов и использование лаконичного YAML вместо сложных скриптов позволяют быстро автоматизировать рутину.

Файл сценария (например, ping.yml) имеет строгую структуру отступов (по 2 пробела, табы запрещены):

---
- name: Тестовая проверка доступности хостов
  hosts: all
  gather_facts: no
  tasks:
    - name: Выполнение модуля ping
      ansible.builtin.ping:

Запуск сценария в работу осуществляется командой: ansible-playbook -i hosts ping.yml.

Инвентарь (Inventory) — конфигурационный файл (в формате INI или YAML), в котором администратор жестко описывает список IP-адресов или доменных имен всех управляемых серверов, группируя их по ролям (например: [web_servers], [db_servers]), а также задает специфичные переменные подключения.

Билет №24

В отличие от обычного прокси (Forward Proxy), защищающего клиентов внутри сети, Reverse Proxy (обратный прокси, например Nginx, HAProxy) устанавливается на границе сети перед серверами бэкенда, защищая и оптимизируя их работу. Клиенты из Интернета думают, что обращаются напрямую к приложению, а фактически коммутируют с реверс-прокси.

Решаемые задачи:
  • Балансировка нагрузки (Load Balancing): Прокси принимает миллионы запросов и равномерно распределяет их по пулу внутренних серверов (Round Robin, Least Connections), предотвращая их перегрузку.
  • Терминация SSL/TLS: Прокси берет на себя тяжелую математику расшифровки HTTPS-трафика. Внутрь локальной сети к бэкендам трафик идет уже по быстрому открытому HTTP, разгружая процессоры серверов приложений.
  • Сокрытие инфраструктуры (Безопасность): Злоумышленник снаружи видит только один IP-адрес прокси-сервера и не знает структуру, адреса и ОС серверов бэкенда, находящихся в приватной зоне.

Перенаправление входящих запросов с внешнего порта 80 на внутреннее приложение (например, работающее на Node.js или Python на порту 8080) настраивается в блоке location:

location / {
    proxy_pass http://127.0.0.1:8080;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
}

Директивы proxy_set_header критически важны: без них сервер бэкенда будет думать, что все запросы совершает сам Nginx, и не сможет узнать реальные IP-адреса посетителей.

Балансировка нагрузки (Load Balancing) — метод распределения входящих сетевых запросов или задач между группой вычислительных серверов (аппаратных или виртуальных) с целью оптимизации использования ресурсов, сокращения времени ответа и обеспечения высокой отказоустойчивости сервиса.

Билет №25

PAM (Pluggable Authentication Modules) — подключаемые модули аутентификации. Это гибкая архитектура, разделяющая логику работы программ (SSH, FTP, gdm) и непосредственные методы проверки подлинности пользователей.

Процесс работы:

Когда пользователь пытается войти, например, по SSH, демон sshd не проверяет пароль сам. Он вызывает библиотеку PAM, передавая ей имя сервиса. PAM обращается к конфигурационному файлу /etc/pam.d/sshd. В этом файле прописана цепочка динамических библиотек (модулей `.so`). PAM поочередно выполняет их: один модуль может проверять пароль в локальном /etc/shadow, второй — требовать токен двухфакторной аутентификации (2FA), третий — сверять системные лимиты (время входа). По результатам прохождения цепочки PAM возвращает приложению простой вердикт: `Доступ разрешен` или `Запрещен`.

Конфигурация PAM разделена на 4 независимых типа управления (интерфейса):

  • auth: Проверяет подлинность субъекта. Запрашивает и валидирует пароли, биометрию или криптографические ключи.
  • account: Проверяет законность доступа аккаунта. Выясняет, не истек ли срок действия учетной записи, не заблокирован ли пользователь администратором, разрешен ли вход в этот час суток.
  • password: Отвечает за механизмы обновления паролей. Контролирует сложность нового пароля (длина, символы) через модуль pam_pwquality.
  • session: Настраивает окружение до и после предоставления доступа. Монтирует домашние директории, задает лимиты на ресурсы (ulimit через pam_limits.so), пишет записи в журналы wtmp/utmp.

Kerberos — сетевой протокол взаимной аутентификации, работающий на базе концепции доверенной третьей стороны (KDC). Вместо передачи паролей по сети, он использует симметричное шифрование и выдачу временных электронных билетов (Tickets), реализуя безопасный механизм Single Sign-On.

Билет №26

cron — классический системный демон-планировщик задач в Unix-подобных операционных системах. Его назначение — автоматический запуск регулярных фоновых скриптов, команд и утилит в строго заданные моменты времени.

Раз в минуту демон cron просыпается, считывает конфигурационные файлы таблиц (crontabs) и проверяет, совпадает ли текущая минута, час и дата с условиями запуска задач. Роль cron в автоматизации огромна: на него возлагаются задачи ежедневного резервного копирования, ротации и сжатия старых логов (logrotate), синхронизации времени, очистки кэшей и временных папок (/tmp), а также сбор метрик здоровья серверов.

Конфигурация cron состоит из 5 полей времени и пути к исполняемому файлу.

Для настройки пользователь открывает редактор расписания командой: crontab -e, и добавляет в конец файла следующую строку:

0 3 * * * /usr/local/bin/backup.sh > /dev/null 2>&1 Разбор полей времени:

0 — 0-я минута; 3 — 3-й час ночи; * — любой день месяца; * — любой месяц; * — любой день недели.
Конструкция > /dev/null 2>&1 глушит стандартный вывод команды, чтобы cron не слал локальные почтовые уведомления администратору при каждом запуске.

crontab (cron table) — текстовый конфигурационный файл, содержащий список команд, инструкций и точное расписание времени их выполнения, редактируемый специализированной одноименной командой crontab -e ради автоматической проверки синтаксиса.

Билет №27

СКС (Структурированная кабельная система) — иерархическая кабельная система здания или комплекса зданий, спроектированная по строгим международным стандартам (ISO/IEC 11801).

Она состоит из 3 ключевых подсистем:
  • Магистральная подсистема комплекса (Campuse Backbone): Объединяет кабельными трассами (чаще всего уличным оптоволокном) несколько отдельно стоящих зданий компании с центральной серверной.
  • Магистральная подсистема здания (Building Backbone / Вертикальная): Соединяет центральный главный кросс (ГК) здания с этажными распределительными кроссовыми шкафами. Реализуется многомодовым оптоволокном или многопарной витой парой высоких категорий.
  • Горизонтальная подсистема: Включает в себя кабели витой пары, проложенные от патч-панелей в этажном телекоммуникационном шкафу до информационных розеток на рабочих местах сотрудников. Длина кабеля в горизонтальном сегменте по стандарту строго ограничена **90 метрами** для медной витой пары.

Стандарты регламентируют цветовую схему раскладки 8 проводников витой пары по контактам коннектора RJ-45 (8P8C).

  • T568A: Бело-зеленый, зеленый, бело-оранжевый, синий, бело-синий, оранжевый, бело-коричневый, коричневый.
  • T568B: Бело-оранжевый, оранжевый, бело-зеленый, синий, бело-синий, зеленый, бело-коричневый, коричневый.

Разница заключается в перестановке местами зеленой и оранжевой пар. В СНГ, Европе и коммерческом секторе США повсеместно и доминантно распространен стандарт **T568B**. Для создания прямого патч-корда (Straight-through) кабель жмется с двух сторон по схеме B. Для создания старых перекрестных кабелей (Crossover) один конец жался по A, второй — по B (современное сетевое железо поддерживает технологию Auto-MDIX и само скрещивает пары, поэтому кроссоверы ушли в прошлое).

Патч-панель (коммутационная панель) — пассивное сетевое оборудование, устанавливаемое в 19-дюймовую стойку. С тыльной стороны в неё намертво заделываются (забиваются инструментом) медные кабели горизонтальной подсистемы, идущие от розеток пользователей. С лицевой стороны порты выводятся в виде стандартных гнезд RJ-45. Нужна для гибкой коммутации и защиты стационарного кабеля от изломов.

Билет №28

Физическая безопасность ЦОД/серверной — первый эшелон защиты ИТ-инфраструктуры, предотвращающий физическое уничтожение, кражу или порчу оборудования.

Ключевые принципы организации:
  • Разграничение зон доступа (СКУД): Вход в серверную зону защищается биометрическими шлюзами, считывателями карт доступа. Все перемещения логируются. Серверные стойки закрываются на индивидуальные механические или кодовые замки.
  • Круглосуточное видеонаблюдение: Камеры высокого разрешения должны контролировать периметр, входы в гермозоны и каждый коридор между стойками с хранением архива записей не менее 30 дней.
  • Защита от техногенных аварий: Помещение должно быть оснащено автоматической системой газового пожаротушения (газ хладон или азот тушит огонь, вытесняя кислород, но не повреждает электронику, в отличие от воды или порошка). На полу монтируются датчики протечки воды.

Для стабильной работы микросхем и дисков датчики мониторинга окружающей среды (например, системы NetBotz) отслеживают:

  • Температуру: По стандартам ASHRAE оптимум в "холодном коридоре" составляет 18–27°C. Перегрев приводит к деградации процессоров и уходу серверов в тепловой шатдаун (Thermal Throttling).
  • Относительную влажность: Должна быть в рамках 40–60%. Слишком сухой воздух провоцирует накопление статического электричества (риск пробоя плат искрой), слишком влажный — вызывает микрокоррозию контактов и риск конденсата.
  • Запыленность: Контроль чистоты воздуха, так как пыль забивает радиаторы охлаждения и служит проводником статики.

ИБП (UPS) — высокомощное электронное устройство с блоком аккумуляторных батарей. Используется для фильтрации скачков и просадок входного напряжения электросети, а при полном исчезновении питания — для мгновенного (за 0 миллисекунд) перевода серверов на питание от АКБ на время, достаточное для автоматического запуска дизель-генераторных установок (ДГУ) или корректного программного выключения ОС.

Билет №29

Назначение систем мониторинга (Zabbix, Prometheus, Nagios) — обеспечение непрерывной доступности (Uptime) бизнес-сервисов, проактивное обнаружение аномалий до того, как они приведут к аварии, и сбор метрик для планирования масштабирования (Capacity Planning).

Объекты обязательного контроля:
  • Доступность узлов (Инфраструктурный уровень): Проверка пингов (ICMP) сетевого оборудования, доступности портов критических служб (TCP 22, 80, 443).
  • Ресурсы операционных систем (Уровень хостов): Утилизация процессора (CPU Load), заполненность оперативной памяти (RAM), свободное место на дисковых разделах (особенно логи /var/log и базы /var/lib), IOPS дисков.
  • Сетевые метрики: Пропускная способность интерфейсов (вход/выход Мбит/с), количество ошибок на портах коммутаторов (Errors/Discards), задержки каналов.

Консольный сниффер tcpdump требует root-прав. Команда для перехвата пингов выглядит так:

tcpdump -i eth0 icmp -nn -w /tmp/ping_capture.pcap Разбор флагов команды:
  • -i eth0 — слушать конкретный сетевой интерфейс.
  • icmp — фильтр захвата (BPF), отсекает всё, кроме протокола ICMP.
  • -nn — отключает преобразование IP-адресов в доменные имена и портов в имена служб (экономит CPU и ускоряет работу).
  • -w /tmp/ping_capture.pcap — запрещает печатать пакеты на экран, принудительно сохраняя их в бинарный файл формата PCAP, который в дальнейшем можно открыть в графическом анализаторе Wireshark для детального разбора.
  • ping: Проверяет исключительно конечную связность между вашим хостом и целью по протоколу ICMP (пакеты Echo-Request/Echo-Reply). Показывает факт доступности, процент потерь и среднее время задержки (RTT). Что происходит с пакетом в пути — пингу неизвестно.
  • traceroute: Трассирует и раскрывает полную цепочку промежуточных маршрутизаторов (хопов) на всем пути прохождения пакета к цели. Работает за счет последовательного инкремента поля **TTL (Time to Live)** в заголовке IP (сначала шлет пакет с TTL=1, первый роутер сбрасывает его в 0, уничтожает пакет и шлет обратно `ICMP Time Exceeded`, раскрывая свой IP; затем шлет TTL=2 и т.д.), фиксируя задержки до каждого транзитного узла.
